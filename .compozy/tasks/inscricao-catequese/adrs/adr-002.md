# ADR-002: Formulário Único com Flag de Renovação

## Status

Accepted

## Date

2026-05-27

## Context

O formulário público de inscrição precisa atender tanto alunos novos quanto renovações de matrícula de alunos que já frequentam a catequese. Precisamos decidir se teremos fluxos separados ou um fluxo unificado.

## Decision

Implementar um formulário único em `/inscricao` que atende ambos os cenários. Um campo "Já frequentou a catequese aqui?" adapta o fluxo — quando marcado, solicita o nome anterior para que o admin vincule ao cadastro existente na aprovação. O admin vê uma lista unificada de inscrições com status (pendente/aprovada/rejeitada).

## Alternatives Considered

### Alternative 1: Dois Fluxos Separados (Novo vs Renovação)

- **Description**: Rotas distintas `/inscricao/novo` e `/inscricao/renovacao`. Na renovação, o pai busca o filho por nome + data de nascimento e confirma dados pré-carregados.
- **Pros**: Renovação mais rápida para o pai, dados pré-carregados.
- **Cons**: Escopo maior, expõe dados de alunos existentes publicamente (risco LGPD), duas páginas para manter.
- **Why rejected**: Complexidade desproporcional ao MVP e risco de exposição de dados pessoais.

### Alternative 2: Formulário Único sem Distinção

- **Description**: Todos preenchem o mesmo formulário completo, sem indicação de novo ou renovação.
- **Pros**: Mais simples possível de implementar.
- **Cons**: Pais de renovação preenchem tudo de novo, mais trabalho para o admin identificar duplicatas.
- **Why rejected**: Experiência ruim para renovações e sobrecarga desnecessária no admin.

## Consequences

### Positive

- Uma única página pública para manter
- Fluxo simples e familiar para o pai
- Admin tem contexto sobre renovações sem complexidade técnica

### Negative

- Renovações dependem do admin identificar e vincular manualmente ao aluno existente
- Campo "já frequentou" é autodeclarado e pode ser impreciso

### Risks

- Admin pode não encontrar o aluno existente pelo nome informado — mitigação: busca flexível no painel

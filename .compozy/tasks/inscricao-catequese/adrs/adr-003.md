# ADR-003: Tabela Enrollments Separada de Students

## Status

Accepted

## Date

2026-05-27

## Context

O formulário público de inscrição coleta dados do catequizando, mas a turma só é atribuída pelo admin na aprovação. A tabela `students` tem `class_id NOT NULL`, então não é possível inserir diretamente sem turma. Precisamos decidir onde armazenar as inscrições pendentes.

## Decision

Criar uma tabela `enrollments` separada que armazena as inscrições com status (`pending`, `approved`, `rejected`). Na aprovação, o admin seleciona a turma e o sistema cria o registro em `students` (ou vincula a um existente no caso de renovação), atualizando o enrollment com `approved_student_id` e `approved_class_id`.

## Alternatives Considered

### Alternative 1: Inserir diretamente em Students com class_id nullable

- **Description**: Alterar a constraint de `class_id` para nullable e adicionar uma coluna de status na tabela `students`.
- **Pros**: Sem tabela adicional, menos duplicação de colunas.
- **Cons**: Quebra a integridade da tabela `students` (alunos sem turma), mistura inscrições pendentes com alunos ativos, requer refatorar queries existentes que assumem `class_id NOT NULL`.
- **Why rejected**: Alto risco de regressão nas funcionalidades existentes e violação do modelo de dados estabelecido.

## Consequences

### Positive

- Tabela `students` mantém integridade existente intacta
- Inscrições pendentes ficam isoladas do fluxo de alunos ativos
- Fácil de adicionar campos específicos de inscrição (is_renewal, previous_name, rejection_reason) sem poluir `students`

### Negative

- Duplicação de colunas entre `enrollments` e `students`
- Na aprovação, dados são copiados de uma tabela para outra

### Risks

- Dados podem ficar inconsistentes entre enrollment e student criado — mitigação: copiar todos os campos na mesma transaction

## Implementation Notes

A tabela `enrollments` referencia `academic_year_id` para vincular ao ano letivo, e `approved_student_id` / `approved_class_id` para rastrear o resultado da aprovação.

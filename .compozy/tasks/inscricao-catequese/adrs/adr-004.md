# ADR-004: Server Action com Service Role para Inserção Pública

## Status

Accepted

## Date

2026-05-27

## Context

O formulário público de inscrição precisa inserir dados na tabela `enrollments` sem autenticação do usuário. O Supabase usa RLS (Row Level Security) que por padrão bloqueia operações do role `anon`. Precisamos decidir como permitir a inserção pública de forma segura.

## Decision

Usar um Server Action do Next.js que chama `createSupabaseAdminClient()` (service role) para inserir na tabela `enrollments`. O service role bypassa RLS, e como o código roda exclusivamente no servidor, a chave secreta nunca é exposta ao cliente. A tabela `enrollments` terá policies RLS apenas para `authenticated` (coordinator/admin) para leitura e atualização.

## Alternatives Considered

### Alternative 1: RLS Policy para role anon

- **Description**: Criar uma policy `INSERT TO anon` na tabela `enrollments`, permitindo inserção anônima via PostgREST.
- **Pros**: Segue o padrão Supabase de controle via RLS, mais granular.
- **Cons**: Abre superfície de ataque direta no banco (qualquer um com a anon key pode inserir), mais complexo de auditar, requer cuidado extra com rate limiting no nível do banco.
- **Why rejected**: Server Action com service role é mais seguro e mais simples, mantendo o controle no código da aplicação onde podemos adicionar validações extras (rate limiting, CAPTCHA futuro).

## Consequences

### Positive

- Chave secreta nunca exposta ao cliente
- Validação Zod acontece no servidor antes da inserção
- Flexibilidade para adicionar proteções extras (rate limiting, CAPTCHA) no Server Action

### Negative

- Bypassa RLS completamente — qualquer bug no Server Action pode permitir operações indevidas
- Dependência do `SUPABASE_SECRET_KEY` estar configurado no ambiente

### Risks

- Server Action público sem rate limiting pode sofrer spam — mitigação: validação rigorosa de campos e CAPTCHA em fase futura

## Implementation Notes

O `createSupabaseAdminClient()` já existe em `lib/supabase/config.ts`. A rota `/inscricao` deve ser adicionada a `PUBLIC_PATHS` em `lib/auth/routing.ts`.

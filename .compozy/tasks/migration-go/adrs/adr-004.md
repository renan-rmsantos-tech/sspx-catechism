# ADR-004: Autenticação por JWT em cookie httpOnly e autorização na camada de aplicação

## Status

Accepted

## Date

2026-06-23

## Context

O Supabase fornecia hoje: autenticação (GoTrue), sessão em cookie (`@supabase/ssr`) e autorização via RLS (`auth.uid()`, `private.is_coordinator()`, `private.is_class_catechist()`). Saindo do Supabase, a SPA (navegador) chamará a API Go diretamente, e tanto a autenticação quanto a autorização precisam de novo mecanismo. A RLS depende de uma GUC (`auth.uid()`) setada a partir do JWT do Supabase — não há equivalente nativo num Postgres puro acessado por um único role de aplicação.

## Decision

- **Autenticação:** a API Go emite um **JWT assinado** com `sub` (user id) e `role` nas claims, gravado em **cookie httpOnly + Secure + SameSite=Lax**. A SPA não lê o token (XSS não o rouba); o browser o anexa automaticamente às chamadas same-origin `/api`.
- **Autorização:** movida para a **camada de aplicação Go** (`Authorizer` + middlewares `RequireAuth`/`RequireCoordinator`), reimplementando a matriz de políticas RLS (coordenador vê tudo; catequista restrito às suas turmas). A RLS é removida do banco.
- O `role` nas claims evita uma consulta ao banco por request para a checagem grosseira; checagens finas por turma consultam `class_catechists`.

## Alternatives Considered

### Alternative 1: JWT em header (localStorage)

- **Description**: Token no corpo da resposta, guardado em localStorage, enviado em `Authorization: Bearer`.
- **Pros**: Sem CSRF; simples.
- **Cons**: Vulnerável a XSS roubar o token.
- **Why rejected**: cookie httpOnly é mais seguro; same-origin via Caddy elimina a fricção de CORS.

### Alternative 2: Sessão opaca no banco

- **Description**: Token aleatório numa tabela `sessions` + cookie httpOnly.
- **Pros**: Revogação imediata; role sempre fresco.
- **Cons**: 1 consulta por request; mais estado para gerir.
- **Why rejected**: Desnecessário para a escala; JWT é stateless e suficiente. (Pode ser reavaliado se revogação imediata virar requisito.)

### Alternative 3: Manter RLS no Postgres

- **Description**: Setar `auth.uid()` por request a partir do JWT.
- **Pros**: Reaproveita as policies SQL existentes.
- **Cons**: Requer impersonação por request com um role por usuário, atípico fora do Supabase; acopla autorização ao banco.
- **Why rejected**: Mais simples e testável manter a autorização em Go com um único role de app.

## Consequences

### Positive

- Modelo de auth simples, stateless e same-origin.
- Autorização centralizada, testável por testes unitários.
- Sem dependência da infraestrutura de RLS/roles do Supabase.

### Negative

- JWT não revoga antes de expirar (mitigável com expiração curta + logout limpando cookie).
- A lógica de autorização precisa ser fielmente reimplementada e testada (risco de divergência da RLS).

### Risks

- **CSRF**: mitigado por SameSite=Lax (e endpoints de mutação não navegacionais); reforço com verificação de Origin se necessário.
- **Divergência da autorização**: mitigada por testes cobrindo toda a matriz de permissões.

## Implementation Notes

- Fluxo `must_change_password` preservado: login retorna a flag; rotas bloqueadas até a troca.
- Bootstrap idempotente do admin no startup (substitui `instrumentation.ts`).
- Criação de catequista: senha gerada (mesmo formato memorável atual), hash bcrypt, `must_change_password=true`, texto puro exibido uma vez.

# ADR-003: Stack do backend Go — chi + pgx + sqlc, migrações com goose

## Status

Accepted

## Date

2026-06-23

## Context

O backend será reescrito em Go, substituindo o Supabase (Auth + Postgres + PostgREST). O modelo de dados tem joins específicos (`classes(name)`, `attendance_records(...)`), buscas `ILIKE`, upserts idempotentes (`ON CONFLICT`) e mapeamento de códigos de erro PG para HTTP. Era preciso escolher roteador HTTP, camada de acesso ao banco e ferramenta de migração.

## Decision

- **Roteador:** `chi` (leve, idiomático, compatível com `net/http`).
- **Driver/pool:** `pgx`.
- **Acesso a dados:** `sqlc` gerando código Go type-safe a partir de SQL escrito à mão (queries explícitas; upserts e joins permanecem em SQL legível).
- **Migrações:** `goose` (migrações `.sql` versionadas em `backend/db/migrations/`, aplicadas no startup da API ou via CLI).
- **Hash de senha:** `bcrypt` (`golang.org/x/crypto/bcrypt`).

## Alternatives Considered

### Alternative 1: net/http puro + pgx (sem chi, sem sqlc)

- **Description**: Apenas stdlib + queries manuais.
- **Pros**: Mínimo de dependências.
- **Cons**: Mais boilerplate de roteamento e de scan de linhas.
- **Why rejected**: chi+sqlc reduzem boilerplate sem esconder o SQL.

### Alternative 2: Gin + GORM

- **Description**: Framework Gin + ORM GORM.
- **Pros**: Baterias inclusas, familiar.
- **Cons**: GORM abstrai o SQL; os upserts/joins/erros-PG específicos ficam mais limpos em SQL puro.
- **Why rejected**: Menos controle sobre o SQL crítico (sync idempotente, joins).

### Alternative 3: argon2id para hash; golang-migrate para migrações

- **Description**: Variantes equivalentes.
- **Pros**: argon2id mais resistente a GPU; golang-migrate é popular.
- **Cons**: argon2id exige tunar parâmetros e é mais pesado; golang-migrate equivale a goose sem ganho aqui.
- **Why rejected**: bcrypt é suficiente e simples; goose foi a preferência.

## Consequences

### Positive

- SQL explícito e type-safe (sqlc) para as queries críticas; alto desempenho (pgx).
- Migrações embarcadas no binário Go, sem Node no servidor.
- Dependências enxutas e idiomáticas.

### Negative

- sqlc exige reexecutar a geração ao mudar queries (passo de build).
- SQL escrito à mão (vs ORM) para cada operação.

### Risks

- **Reimplementar a autorização que era RLS em Go**: risco de divergência — mitigado por testes unitários cobrindo a matriz de permissões e testes de integração.

## Implementation Notes

- Preservar a leniência de UUID (hex não-RFC4122) da validação atual (`zPgUuid`).
- Sync de presença: `INSERT ... ON CONFLICT (class_id,date) DO NOTHING RETURNING id` + records `ON CONFLICT (session_id,student_id)`; `catechist_id` vem do token.
- Manter o trigger `validate_class_date_day` no banco (defesa em profundidade).

# ADR-001: Hospedagem em VPS único com Docker Compose (backend Go, frontend Next.js)

## Status

Accepted

## Date

2026-06-23

## Context

O sistema de catequese roda hoje em Vercel (frontend Next.js 16 / React 19, com Server Components e Server Actions) + Supabase (Auth, Postgres, RLS, PostgREST). O motivador principal da migração é **redução e previsibilidade de custo**. Restrições e premissas levantadas no brainstorming:

- Os dados pessoais (de menores e responsáveis) **podem ficar em servidor no exterior** — não há exigência formal de residência LGPD no Brasil.
- O próprio desenvolvedor assumirá a **operação completa** (SO, backups, restauração, monitoramento), aceitando auto-hospedar o Postgres.
- O sistema **ainda não está em produção** — não há dados reais a migrar.
- O backend será **reescrito em Go**; o frontend **permanece em Next.js/React** como camada de cliente consumindo a API Go.
- Supabase Storage, Realtime e Edge Functions **não são usados** hoje — apenas Auth + Postgres + RLS precisam de substituto.

## Decision

Hospedar toda a aplicação em **um único VPS de baixo custo (classe Hetzner CPX22, ~€8/mês)** orquestrado por **Docker Compose**, com os serviços:

- **Caddy** como reverse proxy com SSL automático (Let's Encrypt);
- **Next.js** como container de frontend (runtime Node);
- **API em Go** como binário compilado em container mínimo;
- **Postgres** auto-hospedado em container;
- **Backup**: `pg_dump` noturno cifrado enviado para armazenamento de objetos off-site (ex.: Cloudflare R2 ou Hetzner Storage Box) + snapshots do provedor.

Custo-alvo: **~$10–15/mês fixo**.

## Alternatives Considered

### Alternative 1: VPS (Go + Postgres) + frontend em CDN gratuita (Cloudflare Pages)

- **Description**: Separar o frontend para um host estático/edge gratuito, deixando só Go + Postgres no VPS.
- **Pros**: CDN global grátis, servidor mais leve, frontend escala sozinho.
- **Cons**: Next.js com recursos dinâmicos não exporta como estático puro — exigiria runtime Node de qualquer forma ou voltar a um host tipo Vercel; mais peças, CORS entre domínios.
- **Why rejected**: Conflita com a decisão de manter Next.js como frontend e adiciona complexidade sem reduzir custo de forma relevante.

### Alternative 2: AWS minimalista (Lightsail / EC2 + RDS) em São Paulo

- **Description**: Usar ecossistema AWS com região no Brasil e Postgres gerenciado (RDS).
- **Pros**: Região no Brasil (latência, conforto LGPD), RDS gerenciado remove ops do banco, ecossistema maduro.
- **Cons**: ~$25–55/mês (3–5× o custo do VPS), preços com pegadinhas (egress, burst); App Runner em modo de manutenção.
- **Why rejected**: Vai diretamente contra o motivador de custo; residência no Brasil não é requisito e o desenvolvedor aceita operar o banco.

## Consequences

### Positive

- Menor custo possível e totalmente previsível (~$10–15/mês).
- Controle total sobre stack, dados e deploy; sem lock-in de Vercel/Supabase.
- Topologia simples: uma máquina, um `docker-compose.yml`.

### Negative

- Ponto único de falha (single box); sem alta disponibilidade nativa.
- Operação (patches de SO, segurança, monitoramento) é responsabilidade do desenvolvedor.
- Latência maior para usuários no Brasil se o servidor estiver na Europa (aceitável para um app de formulários/chamada).

### Risks

- **Perda de dados por falha do disco**: mitigada por `pg_dump` noturno off-site cifrado + snapshots do provedor; restauração testada periodicamente.
- **Comprometimento de segurança**: mitigada por firewall, atualizações automáticas de segurança, acesso só por SSH com chave, Caddy gerenciando TLS.
- **Sobrecarga inesperada**: mitigada pela facilidade de fazer upgrade vertical do VPS (mais vCPU/RAM) em minutos.

## Implementation Notes

- Substituir Supabase Auth por autenticação própria em Go (hash de senha + JWT/sessão), preservando os fluxos atuais: login por e-mail/senha, troca obrigatória de senha (`must_change_password`), criação de catequista pelo coordenador, seed do admin no startup.
- Mover a autorização baseada em RLS (`is_coordinator`, `is_class_catechist`) para a camada de aplicação em Go.
- Reescrever as chamadas PostgREST (`.from()`) e os endpoints `app/api/**` como handlers Go; preservar a idempotência do sync de presença offline.
- Manter o runner de migrações SQL (`pg` → equivalente em Go ou ferramenta de migração Go) apontando para o novo Postgres.
- Next.js passa a chamar a API Go via HTTP; Server Actions viram chamadas à API.

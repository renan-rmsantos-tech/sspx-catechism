# ADR-002: Frontend como SPA Vite/React consumindo a API Go

## Status

Accepted

## Date

2026-06-23

## Context

O frontend atual é Next.js 16 com Server Components e Server Actions que leem/escrevem direto no Postgres via Supabase (cookie-bound). Com o backend indo para uma API Go autônoma (única fonte de dados), os Server Components/Actions perdem o acesso direto ao banco. O PRD pediu "manter o front em React". Era preciso decidir a topologia frontend↔backend e o que fazer com o Next.js, que sem SSR/Server Actions perde seu propósito principal.

## Decision

Migrar o frontend para uma **SPA estática com Vite + React + React Router**, servida como arquivos estáticos pelo Caddy. O navegador chama a API Go diretamente em `/api/*` (`fetch` com `credentials: 'include'`). Reaproveitam-se componentes, Tailwind/shadcn e o PWA (via `vite-plugin-pwa`). O Go passa a ser a única fonte de dados e de autorização.

## Alternatives Considered

### Alternative 1: Next.js como cliente puro da API Go

- **Description**: Manter Next.js, mas Server Components/Actions chamam a API Go via HTTP.
- **Pros**: Preserva mais código atual e o roteador do Next; mantém SSR.
- **Cons**: Exige runtime Node sempre-ligado no servidor (mais um container/memória), e adiciona um hop servidor→Go nas leituras; Next.js sem acesso a dados direto perde grande parte do valor.
- **Why rejected**: O usuário optou por simplicidade e por remover o runtime Node do servidor.

### Alternative 2: Next.js em static export (`output: 'export'`)

- **Description**: Next.js gerando SPA estática só com client components.
- **Pros**: Preserva o roteador do Next e o Serwist; sem runtime Node.
- **Cons**: Mantém o peso/configuração do Next.js para um uso que ele não foi feito; migração de Server Components para client é trabalho similar ao de ir para Vite.
- **Why rejected**: Vite é mais leve e direto para uma SPA pura; o usuário preferiu Vite.

## Consequences

### Positive

- Hospedagem trivial do frontend (arquivos estáticos no Caddy), sem runtime Node no servidor.
- Arquitetura mais simples: uma fonte de dados (Go), um modelo de auth (JWT em cookie).
- Build mais rápido e bundle menor que Next.js.

### Negative

- Perde SSR (SEO/first-paint) — irrelevante para um app interno autenticado.
- Exige reescrever as telas que eram Server Components para buscar dados via cliente HTTP.

### Risks

- **Regressão de fluxos ao reescrever telas**: mitigada pela suíte Vitest reaproveitada + checklist manual dos 3 perfis.
- **CORS/cookies**: evitado servindo SPA e API sob o mesmo domínio via Caddy (mesma origem).

## Implementation Notes

- React Router substitui o roteamento de arquivos do Next e o `proxy.ts` (guarda de rotas por papel no cliente, com o servidor sempre validando o JWT).
- PWA/offline (Dexie, fila de sync, background sync) portado para `vite-plugin-pwa`; nenhuma parte do offline dependia de Supabase.

# ADR-005: Empacotamento Docker Compose, deploy via GitHub Actions e backup no Hetzner Storage Box

## Status

Accepted

## Date

2026-06-23

## Context

A Abordagem A (ADR-001) define um VPS Hetzner único. Era preciso detalhar o empacotamento dos serviços, a estratégia de TLS, o pipeline de deploy e a rotina de backup off-site exigida pelo PRD (backup automático + restauração testada), além da organização do código (monorepo vs múltiplos repos).

## Decision

- **Organização:** **monorepo** com `frontend/` (Vite), `backend/` (Go) e, na raiz, `docker-compose.yml`, `Caddyfile` e `.github/workflows/`.
- **Empacotamento:** Docker Compose com 4 serviços — `caddy` (reverse proxy + TLS automático, único exposto: 80/443), `api` (Go, multi-stage build), `db` (Postgres 16, volume nomeado, sem porta exposta) e build estático do `frontend` servido pelo Caddy. Um serviço/cron de backup.
- **TLS:** Caddy via ACME/Let's Encrypt (automático).
- **Deploy:** **GitHub Actions** no push à `main` → build das imagens + `docker compose up -d` no VPS via **SSH** (chave em secret).
- **Backup:** `pg_dump` noturno cifrado → **Hetzner Storage Box** via SFTP, com retenção; restauração testada e documentada.

## Alternatives Considered

### Alternative 1: Dois repositórios separados

- **Description**: Repos independentes para frontend e backend.
- **Pros**: Isola ciclos de vida.
- **Cons**: Coordenação extra de versões/deploy para um time de uma pessoa.
- **Why rejected**: Monorepo simplifica deploy e versionamento unificados.

### Alternative 2: Deploy manual via script SSH / Registry + Watchtower

- **Description**: `deploy.sh` manual, ou imagens no GHCR auto-atualizadas por Watchtower.
- **Pros**: Manual é simples; Watchtower automatiza sem CI.
- **Cons**: Manual é repetitivo; Watchtower tira o controle do momento do deploy e adiciona peças.
- **Why rejected**: GitHub Actions→SSH equilibra automação e controle.

### Alternative 3: Backup no Cloudflare R2 / Backblaze B2

- **Description**: Object storage S3-compatível de terceiros.
- **Pros**: Independente do provedor do VPS; R2 sem egress.
- **Cons**: Mais uma conta/credencial; benefício marginal aqui.
- **Why rejected**: Hetzner Storage Box mantém tudo num fornecedor e é barato (~€3,2/mês/1TB).

## Consequences

### Positive

- Topologia reproduzível: subir o ambiente do zero com um comando.
- Custo previsível (~$10–15/mês) e tudo num fornecedor.
- TLS e deploy automáticos.

### Negative

- Backup no mesmo fornecedor (Hetzner) — risco de correlação de falha do fornecedor (mitigável com cópia adicional se desejado).
- Ponto único de falha no VPS (aceito no ADR-001).

### Risks

- **Perda de dados**: mitigada por `pg_dump` noturno off-site cifrado + restauração testada + snapshots do provedor.
- **Segredo SSH/CI vazado**: mitigado por chave dedicada de deploy, escopo mínimo e firewall.

## Implementation Notes

- Postgres nunca exposto à internet (somente rede interna do Compose).
- `.env` no servidor: `DATABASE_URL`, `JWT_SECRET`, `ADMIN_EMAIL/PASSWORD`, credenciais do Storage Box.
- Runbook de provisionamento do Hetzner documentado no TechSpec (Development Sequencing).
